# "AI에게 앱 만들어달라고 했는데, 더 잘하는 방법이 있을까?"

> **분석 대상**: 영수증 지출관리 앱을 AI와 함께 개발하면서 주고받은 프롬프트 기록 (`prompt.md`)
>
> **분석 프레임워크**: 두 가지 실전 검증된 AI 개발 방법론을 기준으로 분석
> - **CAOF** — AI에게 일을 시킬 때 "역할을 나눠서" 시키는 방법
> - **경량 SDD** — AI에게 "뭘 만들어야 하는지" 명확하게 알려주는 방법
>
> 작성일: 2026-04-07

---

## 먼저, 두 가지 방법론을 쉽게 설명하면

### CAOF가 뭔가요? — "한 사람에게 다 시키지 마라"

회사에서 신입에게 "이거 조사하고, 설계하고, 만들고, 테스트까지 다 해"라고 하면 어떻게 될까? 높은 확률로 조사를 대충 하고, 설계를 건너뛰고, 일단 만들다가 버그가 나면 땜빵하게 된다.

AI도 똑같다. **하나의 AI에게 모든 걸 한꺼번에 시키면, 분석을 건너뛰고 일단 코드부터 쓴다.** 그래서 잘 돌아가는 것 같지만 근본적인 문제가 남아있고, 나중에 더 큰 버그로 터진다.

CAOF는 이 문제를 **역할 분리**로 해결한다:

```
[분석가] "이 버그의 원인은 A 파일의 B 함수에서 null 처리가 빠져있기 때문이야"
    ↓
[구현자] "알겠어, B 함수에 null 체크를 추가할게"
    ↓
[검수자] "원래 의도대로 동작하는지 확인해볼게"
```

한 사람이 다 하는 게 아니라, **진단하는 사람 따로, 고치는 사람 따로, 확인하는 사람 따로**. 이렇게 하면 "일단 고쳐보자"식의 땜빵이 줄어든다.

추가로, CAOF는 **작업의 위험도에 따라 프로세스 무게를 다르게** 한다:

| 위험도 | 예시 | 프로세스 |
|--------|------|----------|
| **가벼움** | 오타 수정, 색상 변경 | 바로 수정 |
| **보통** | 기존 기능 수정, 버그 수정 | 원인 분석 → 수정 |
| **무거움** | 새 기능, 외부 API 연동 | 조사 → 설계 → 승인 → 구현 → 검수 |

오타 하나 고치는데 회의를 열 필요는 없지만, 결제 시스템을 새로 만드는데 "일단 만들어봐"는 위험하다는 거다.

---

### 경량 SDD가 뭔가요? — "끝난 건지 안 끝난 건지 한눈에 보이게"

AI에게 "로그인 기능 만들어줘"라고 하면, AI는 나름대로 만든다. 그런데:
- 소셜 로그인도 포함인가?
- 비밀번호 찾기도?
- 에러 메시지도 한글화?
- **"다 됐어요"라고 했는데, 내가 원한 것과 다르면?**

이런 문제를 막기 위해, **"이게 되면 완료"라는 체크리스트를 미리 만들어두는 것**이 경량 SDD의 핵심이다.

```markdown
## 로그인 기능 — 수락 기준

- [x] 이메일+비밀번호 로그인 동작
- [x] Google 소셜 로그인 동작
- [ ] 비밀번호 찾기 이메일 발송    ← 아직 안 됨
- [ ] 로그인 실패 시 한글 에러 메시지
```

이 체크리스트만 보면:
- **진행률**: 2/4 완료 (50%)
- **남은 일**: 비밀번호 찾기, 에러 메시지
- **별도의 "상태 관리"가 필요 없음** — 체크 표시 자체가 상태

핵심 통찰: **"status: 진행중" 같은 별도 상태 필드는 결국 아무도 안 바꾼다.** 코드를 짜면서 체크박스를 치는 건 자연스럽지만, 별도 상태를 업데이트하는 건 잊어버린다.

## 이제 prompt.md를 분석해보자

### 어떤 프로젝트인가

영수증 사진을 찍으면 → OCR로 텍스트 추출 → AI가 항목별로 정리 → 지출 통계까지 보여주는 앱.
기술 스택: FastAPI(백엔드) + React(프론트엔드) + Upstage OCR + Solar LLM

PRD(기획서)에 개발 단계가 나뉘어 있고, 단계별로 "이거 시작해줘"라고 AI에게 시키는 방식으로 진행했다.

---

## 잘 하고 있는 4가지

### 1. 단계별로 나눠서 진행하고 있다

```
1단계: 백엔드 환경 구성 → OCR 연동 → LLM 연동
2단계: React UI 구성 → 조회/수정/삭제
3단계: 통계 대시보드
```

한 번에 "앱 만들어줘"가 아니라 단계별로 시키는 건 좋은 습관이다.

### 2. 스크린샷으로 버그를 설명한다

```
@images/ui_image_깨짐_01.png
@images/ui_image_깨짐_02.png
```

"안 돼요"가 아니라 **실제 화면을 보여주면서** 문제를 설명하고 있다. AI가 문제를 정확히 파악하는 데 큰 도움이 된다.

### 3. 도메인 지식을 정확히 전달한다

```
"뚜뚜호(A-50)은 물품명이 아니라 요청한 사람의 닉네임 같습니다"
"동원) 소화잘되는초코는 증정 품목이므로 금액을 0으로 보여줘야 할 것 같아요"
```

AI가 모를 수 있는 **영수증 도메인의 맥락**(닉네임 vs 물품명, 증정 품목의 의미)을 구체적으로 알려주고 있다.

### 4. 기능 추가 후 문서를 갱신한다

```
"추가된 기능을 PRD문서에도 반영해서 갱신해 줘"
```

만든 기능을 기획서에 다시 반영하는 습관은 문서와 실제 코드가 따로 노는 걸 방지한다.

## 개선하면 좋은 5가지

### 문제 1: "다 됐어"의 기준이 없다

**현재:**
```
1단계 OCR 연동 구현을 시작해 줘
```

AI: (나름대로 만든다) → "구현했습니다!"
사용자: "...테스트는 어떻게 해요?"
AI: (추가 설명)
사용자: (테스트해보니 500 에러)
AI: (수정)

**이 과정이 비효율적인 이유:** "OCR 연동"이 뭘 의미하는지 AI와 사람 사이에 합의가 없어서, AI가 만든 결과물이 기대와 다를 수 있고, 빠진 게 있는지 확인할 방법도 없다.

**개선:**
```
1단계 OCR 연동 구현을 시작해 줘

완료 기준:
- [ ] Upstage API 키를 환경변수로 관리
- [ ] /api/receipts/upload에 이미지 올리면 OCR 텍스트 반환
- [ ] 지원 안 하는 파일이면 에러 메시지 반환
- [ ] 테스트하는 방법을 알려줘
- [ ] 서버가 정상 빌드됨
```

**효과:**
- AI가 **어디까지 해야 하는지** 명확히 안다
- 사용자가 **빠진 게 있는지** 체크리스트로 확인할 수 있다
- "테스트 방법도 알려줘"를 나중에 따로 물을 필요가 없다

---

### 문제 2: 새로운 기술을 연동할 때 "일단 만들어"부터 시작한다

**현재:**
```
Upstage Document Parse OCR 연동 구현을 시작해 줘
```

바로 구현부터 들어간다.

**왜 위험한가:** Upstage API가 어떤 형식으로 응답하는지, 에러 코드는 뭐가 있는지, 요금 체계는 어떤지를 AI가 **정확히 알고 있다는 보장이 없다.** AI는 "아마 이렇겠지"라고 추측해서 코드를 짜고, 실제로 돌리면 500 에러가 나는 거다. (실제로 이 프로젝트에서 발생한 일이다.)

**개선:**
```
Upstage Document Parse OCR 연동을 구현하려고 해.
먼저 Upstage API 공식 문서를 확인해서:
1) 요청/응답 형식
2) 필요한 인증 방식
3) 에러 코드
를 정리해줘. 확인 후에 구현을 시작할게.
```

**효과:** 5분의 조사로 1시간의 디버깅을 아낄 수 있다.

---

### 문제 3: 버그가 나면 "원인 분석 없이 바로 수정"을 시킨다

**현재:**
```
500 Internal Server Error
터미널의 Log을 확인하고 에러 원인을 수정해 줘
```

한 문장에 "확인"과 "수정"이 동시에 들어있다.

**왜 위험한가:** AI가 로그를 보고 **가장 먼저 눈에 띄는 증상**만 고치게 된다. 예를 들어:
- 증상: "파일 경로를 못 찾음"
- AI의 즉흥 수정: 경로를 하드코딩
- 실제 원인: 환경변수 설정이 빠짐
- 결과: 다른 환경에서 또 터짐

**개선 — 두 단계로 나누기:**
```
[1단계] 500 에러 발생. 터미널 로그를 분석해서 원인이 뭔지 알려줘. (아직 수정하지 마)
```
→ AI가 "원인은 X입니다. 수정 방향은 A 또는 B가 있습니다"라고 보고

```
[2단계] A 방향으로 수정해줘.
```

**효과:** 
- 수정 전에 원인을 이해할 수 있다
- 여러 수정 방향 중 선택할 수 있다
- 땜빵이 아닌 근본 수정이 된다

---

### 문제 4: 모든 작업을 같은 무게로 처리한다

현재 프롬프트에서 아래 작업들이 전부 "구현해 줘" 한 마디로 시작된다:

| 작업 | 실제 복잡도 | 실패하면? |
|------|-----------|----------|
| Langchain 버전 변경 | 낮음 | requirements.txt 되돌리면 끝 |
| OCR API 연동 | **높음** | API 키, 요청 형식, 에러 처리 다 엮임 |
| LLM JSON 구조화 | **높음** | 프롬프트 설계, 파싱 로직, 예외 처리 |
| UI 텍스트 수정 | 낮음 | 한 줄 고치면 끝 |

**왜 문제인가:** 오타 수정과 외부 API 연동에 같은 프로세스를 적용하면, 간단한 건 과하고 복잡한 건 부족하다.

**개선 기준:**

```
[바로 시키기] 오타, 색상, 텍스트 수정 등 → "이거 고쳐줘"
[분석 먼저]  기존 기능 수정, 버그 수정 → "원인 먼저 분석해줘"
[조사부터]   새 API 연동, 새 기능 → "먼저 조사하고, 설계 보여주고, 그다음 구현해"
```

---

### 문제 5: PRD 하나에 모든 게 섞여 있다

현재 PRD 문서가 기획서 + 작업 지시서 + 진행 추적 + 변경 이력을 전부 담당하고 있다. 프로젝트가 커질수록 이 문서가 비대해지고, "지금 뭘 하고 있는지"를 찾기 어려워진다.

**개선 — 기능별 스펙 파일 분리:**

```
specs/
├── _template.md            # 템플릿
├── ocr-integration.md      # OCR 연동 (체크 4/5 → 거의 완료)
├── llm-structuring.md      # LLM 구조화 (체크 3/3 → 완료!)
├── receipt-image-view.md   # 원본 이미지 보기 (체크 1/3 → 진행 중)
└── dashboard-charts.md     # 통계 대시보드 (체크 0/4 → 시작 전)
```

PRD는 "전체 기획"으로 유지하고, **각 기능의 구현 추적은 별도 파일**에서 한다. 파일 하나만 열면 해당 기능이 어디까지 됐는지 바로 보인다.

## 실전 적용 — "다음에 AI에게 시킬 때 이렇게 해보세요"

### Before (현재 방식)

```
@PRD 9.개발 우선순위에서 1단계 OCR 연동 구현을 시작해 줘
구현 후에 테스트 하는 방법도 알려 줘
```

→ 이후 500 에러 발생 → "로그 보고 수정해줘" → 수정 → 또 다른 문제 → ...

### After (개선 방식)

```
@PRD 9.개발 우선순위에서 1단계 OCR 연동을 구현하려고 해.

이건 외부 API 연동이라 복잡도가 높으니까 단계적으로 가자:

1단계 - 조사: Upstage Document Parse API 문서를 확인해서
  요청/응답 형식, 인증 방식, 제한사항을 정리해줘.

2단계 - 구현: 조사 결과를 바탕으로 구현해줘.
  완료 기준:
  - [ ] API 키를 .env로 관리
  - [ ] 이미지 업로드 → OCR 텍스트 추출 성공
  - [ ] 잘못된 파일 형식에 대한 에러 처리
  - [ ] 테스트 스크립트 제공
  - [ ] 빌드 성공
```

### 버그가 났을 때

Before:
```
500 에러. 로그 확인하고 수정해줘.
```

After:
```
500 에러가 났어. 터미널 로그를 보고:
1) 에러 원인이 뭔지 분석해줘 (수정은 아직 하지 마)
2) 수정 방향을 제안해줘
```

---

## 요약 — 핵심 3줄

1. **"뭘 만들면 완료인지" 체크리스트를 미리 적어라** — AI도 사람도 "끝"의 기준이 명확해야 헤매지 않는다.
2. **새로운 기술을 붙일 땐 "조사 먼저, 구현 나중"** — 5분의 조사가 1시간의 디버깅을 아껴준다.
3. **버그가 나면 "분석 먼저, 수정 나중"** — "로그 보고 고쳐"보다 "원인부터 알려줘"가 근본 해결로 이어진다.

이 세 가지만 적용해도 AI와 함께 개발하는 효율이 크게 올라간다.
