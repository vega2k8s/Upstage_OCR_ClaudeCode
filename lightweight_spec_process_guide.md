# 경량 스펙 프로세스 — AI 코딩 에이전트와 일할 때 작업을 추적하는 방법

## 이 문서는 누구를 위한 것인가

AI 코딩 에이전트(Claude Code, Cursor, Copilot 등)와 함께 개발하면서, **"지금 뭘 만들고 있고, 어디까지 했는지"**를 추적하고 싶은 개발자.

특히 Jira/Linear 같은 프로젝트 관리 도구가 과하고, 커밋 메시지만으로는 부족하다고 느끼는 1~2명 규모의 팀이나 개인 개발자에게 맞다.

---

## 이 프로세스가 나온 배경

### 시도 1: 구조화된 티켓 시스템

Epic/Ticket/Usecase 3계층, 5단계 상태 전환(`draft → ready → in-progress → review → done`), 권한 분리(사람 vs AI) — 팀 개발 프로세스를 AI 개발에 그대로 적용했다.

**결과**: 69개 문서, 11,000줄. AI에게 명확한 지시를 줄 수 있었지만:
- 코드는 완료됐는데 티켓 상태가 업데이트 안 됨 (80%의 티켓이 실제와 불일치)
- "1 티켓 = 1 PR" 원칙이 실제 구현 단위와 안 맞음
- 유즈케이스 문서를 만들었지만 아무도 안 봄
- 1인 개발에서 권한 분리는 의미 없음

### 시도 2: 왜 실패했는지 분석

핵심 발견: **수동 상태 관리는 반드시 실패한다.** 코드 구현과 상태 갱신이 별개의 작업이면, 사람이든 AI든 상태 갱신을 빼먹는다.

### 시도 3: 대안 조사

Cursor Rules, Aider, SDD(Spec-Driven Development), AGENTS.md, GitHub Copilot Agent 등 11개 접근법을 비교. 
공통 트렌드: **별도 상태 필드 대신, 기존 인프라(Git, 체크리스트)에서 상태를 파생(derive)하는 방향.**

### 결론

```
문제: "상태를 어떻게 정확하게 관리할까?"
발견: "수동 상태 관리는 반드시 실패한다"
해결: "상태 필드를 제거하라. 체크리스트 자체가 상태다."
```

---

## 핵심 원리: 체크리스트 = 상태

```
❌ 기존 방식:
   status 필드가 별도로 존재 → 코드와 동기화 필요 → 누락 → 불일치
   [status: review] ← 이걸 누가 언제 바꾸나? → 아무도 안 바꿈

✅ 이 방식:
   - [x] DB 스키마 생성
   - [x] API 엔드포인트 구현
   - [ ] 프론트엔드 연동      ← 이것만 보면 "2/3 완료"가 자명함
```

- 체크된 항목 0개 → 아직 시작 안 함
- 일부 체크됨 → 진행 중
- 전부 체크됨 → 완료

**왜 작동하는가:**
1. AI가 코드를 구현하면서 항목을 체크하는 건 자연스러운 동작이다. "상태 업데이트"가 아니라 구현의 일부.
2. 상태가 존재하는 곳이 딱 한 군데(스펙 파일)이므로 이중 관리가 불가능하다.
3. "완료 처리해줘"라고 말할 필요가 없다. 체크리스트가 다 차면 끝이다.

## 폴더 구조

```
your-project/
├── specs/
│   ├── _template.md              # 스펙 템플릿
│   ├── auth-social-login.md      # 진행 중 스펙 (미체크 항목 있음)
│   ├── payment-refund.md         # 완료된 스펙 (전부 체크됨)
│   └── map-offline-cache.md
├── CLAUDE.md                     # (또는 .cursorrules, AGENTS.md 등)
└── src/
```

`active/`, `done/` 같은 하위 폴더를 **두지 않는다.** 폴더 이동도 수동 작업이고, 수동 작업은 누락된다.

진행 상황을 확인하고 싶으면:
```bash
# 완료된 스펙 (미체크 항목 없음)
grep -rL "\- \[ \]" specs/*.md

# 진행 중 스펙 (미체크 항목 있음)
grep -rl "\- \[ \]" specs/*.md
```

## 스펙 파일 템플릿

```markdown
# {기능명}

> {이 기능이 해결하는 문제를 한 줄로}

## 배경

왜 이 기능이 필요한지. 사용자 관점에서 설명.

## 수락 기준

- [ ] 기준 1 — 구체적이고 검증 가능한 조건
- [ ] 기준 2
- [ ] 기준 3
- [ ] 빌드 성공

## 설계 메모

구현 방향, 기술 선택, 참고할 기존 코드 등.
처음엔 비어 있어도 됨. AI가 구현하면서 채울 수도 있음.

## 참고

- 관련 스펙: {다른 스펙 파일명}
- 관련 문서: {외부 링크 등}
```

전체 10~30줄. 핵심은 **수락 기준**뿐이고, 나머지는 맥락 제공용이다.

### 왜 이것만으로 충분한가

| 기존 티켓 시스템에 있던 것 | 이 템플릿 | 이유 |
|---|---|---|
| status 필드 | 없음 | 체크리스트가 상태를 대체 |
| 티켓 ID | 없음 | 파일명이 ID (`auth-social-login.md`) |
| 우선순위 필드 | 없음 | 어떤 스펙을 먼저 할지는 사람이 구두로 지시 |
| 담당자 필드 | 없음 | 1~2명 팀에서 불필요 |
| depends-on / blocks | 참고 란에 텍스트 | 강제하지 않고 참고만. 실제 의존성은 코드가 결정 |
| 브랜치 필드 | 없음 | `git branch --contains`로 파생 가능 |
| 유즈케이스 문서 | 수락 기준 | 별도 파일 대신 같은 파일 내 체크리스트로 통합 |

## 워크플로우

### 사람이 하는 것

```
1. 스펙 파일 작성 (또는 AI에게 초안 작성 요청)
2. "이 스펙 구현해줘" 지시
3. 결과물 확인
```

### AI가 하는 것

```
1. 스펙 파일 읽기
2. 설계 메모 보강 (필요 시)
3. 코드 구현
4. 구현 완료된 수락 기준 항목 체크 ✅
5. 커밋 (스펙 파일명을 커밋 메시지에 포함)
```

### 커밋 컨벤션

```
{type}({스펙파일명}): {설명}

예시:
feat(auth-social-login): Google OAuth 연동 구현
fix(payment-refund): 환불 금액 계산 오류 수정
```

`git log --grep="auth-social-login"`으로 해당 기능의 전체 이력을 추적할 수 있다.

### 스펙이 필요 없는 경우

모든 작업에 스펙이 필요하지 않다.

| 스펙 필요 | 스펙 불필요 (커밋 메시지로 충분) |
|---|---|
| 새 기능 (수락 기준 3개 이상) | 버그 수정 |
| 여러 파일에 걸친 리팩토링 | 스타일 조정 |
| 설계 판단이 필요한 작업 | 인프라/설정 변경 |
| | 1~2줄 수정 |

## 실제 예시

운동 앱에서 시계열 메트릭 수집 기능을 개발하면서 쓴 스펙:

```markdown
# 운동 시계열 메트릭 수집 및 조회

> 운동 중 시간/거리 기반으로 속도·거리 데이터를 샘플링하여 서버에 저장

## 배경

기존에는 운동 결과가 요약값만 저장됐다. 시계열 데이터를 수집하면
속도 변화 그래프, 구간별 분석이 가능해진다.

## 수락 기준

### 데이터 수집
- [x] MetricsCollector 구현 — 시간 기준(5초) + 거리 기준(1m) 샘플링
- [x] Basic Training에 collector 연동
- [x] Interval Training에 collector 연동
- [x] Course Training에 collector 연동

### API 연동
- [x] WorkoutDetail 모델 생성
- [x] History POST 시 메트릭 데이터 포함하여 전송
- [x] 전송 실패 시 메트릭 제거 후 History만 재시도 (fallback)

### 디버그 확인 (현 단계)
- [ ] History 조회 시 메트릭 포함하여 파싱
- [ ] 히스토리 상세에서 디버그 위젯으로 메트릭 표시

### 유저 시각화 (추후)
- [ ] 속도-시간 그래프 표시
- [ ] 속도-거리 그래프 표시

## 설계 메모

- 기존 측정 로직을 수정하지 않고 별도로 스냅샷만 수집
- API: History POST에 workoutDetail 필드를 끼워 보내면 서버가 자동 생성
```

이 파일 하나로 "어디까지 했고, 뭐가 남았는지"가 즉시 보인다. 체크 비율만 봐도 진행률을 알 수 있다.

## 프로젝트에 도입하기

### Step 1: 폴더 생성 + 템플릿 복사 (5분)

```bash
mkdir specs
# 위 템플릿을 specs/_template.md로 저장
```

### Step 2: AI 설정 파일에 규칙 추가

CLAUDE.md (또는 .cursorrules, AGENTS.md 등)에 추가:

```markdown
## 스펙 기반 개발

- `specs/` 폴더에 기능별 스펙 파일이 있다.
- 스펙 구현 요청 시: 스펙 읽기 → 구현 → 수락 기준 체크(✅) → 커밋
- 커밋 메시지: `{type}({스펙파일명}): {설명}`
- status 필드 없음. 체크리스트의 체크 상태가 곧 진행률이다.
- 전부 체크되면 완료. 작은 작업은 스펙 없이 바로 커밋.
```

### Step 3: 첫 스펙 작성해보기

지금 진행 중인 기능 하나를 골라 스펙으로 작성. AI에게 "이 기능 코드를 분석해서 스펙으로 정리해줘"라고 시켜도 된다.

### 이후

```
Week 1: 기능 2~3개를 스펙으로 작성하고 구현 → 수락 기준 작성 감각 습득
Week 2: 체크리스트가 너무 세밀한가? 너무 뭉뚱그려졌는가? → 적절한 입도 찾기
```

## 기존 도구와 공존

### 설계 문서가 이미 있는 경우 (spec/, docs/ 등)

```
spec/    ← 설계 문서 (장기 참조) — "이 기능은 어떻게 동작해야 하는가"
specs/   ← 실행 스펙 (작업 추적) — "지금 뭘 만들고 있는가"
```

역할이 다르다. 설계 문서는 완성 후에도 참조 가치가 있고, 실행 스펙은 체크가 끝나면 역할 종료. 충돌하지 않는다.

### 티켓/이슈 시스템이 이미 있는 경우

GitHub Issues나 Linear를 쓰고 있다면, 스펙 파일은 "AI가 읽을 수 있는 상세 지시서" 역할로 공존 가능. Issue에 스펙 파일 경로를 링크하면 된다.

## 이 프로세스가 적합한/부적합한 상황

### 최적

- 1인 개발 + AI
- 소규모 팀 (1~2명)
- 빠른 이터레이션
- 사이드 프로젝트부터 중규모 앱까지

### 부적합

- **3명 이상 팀**: 담당자 추적이 필요 → Linear/GitHub Issues 사용
- **엄격한 의존성 관리**: A 완료 전 B 시작 불가 같은 강제가 필요한 경우
- **감사(audit) 추적**: 누가 언제 승인했는지 기록이 필요한 규제 환경
